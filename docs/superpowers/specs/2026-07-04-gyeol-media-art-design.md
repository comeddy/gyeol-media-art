# 결 · GYEOL — 제너러티브 미디어 아트 전시 사이트 설계

- 날짜: 2026-07-04
- 상태: 사용자 승인 완료 (콘텐츠·아키텍처·검증/배포 3개 섹션 모두 승인)
- 참고 사이트: https://d3acb4zouf8s0e.cloudfront.net/ (Generative Hours, 11관 46작품, vibe coding 3일)

## 1. 목적과 성공 기준

- **목적**: 사용자 본인의 독창적인 미디어 아트 전시 사이트를 만들어 실제 운영한다.
- **차별화**: 참고 사이트가 알고리즘(Lenia, Physarum 등)을 테마로 삼은 것과 달리, 본 전시는 **한국적 미감을 테마로 삼고 알고리즘을 도구로 사용**한다.
- **성공 기준**:
  1. 8관 32작품이 배포된 URL에서 전부 동작 (콘솔 에러 0, 빈 캔버스 0)
  2. 초기 로딩 전송량 < 100KB (폰트·CDN 라이브러리 제외), 작품 60fps 목표(최소 30fps)
  3. 참고 사이트(3일)보다 빠른 당일 완성
  4. 배포 전 content-review-agent 품질 점수 85점 이상 (환경 필수 규칙)

## 2. 전시 콘텐츠

**전시 제목**: 「결 · GYEOL」 — 물결·나뭇결·숨결·비단결의 '결'. 자연스러운 무늬와 흐름이라는 한국 미감의 본질을 담는다.

**언어**: 한국어 기본 (참고 사이트와 동일하게 영문 포인트 텍스트 병용).

### 인터랙션 분배

- 기본(전 작품): 마우스/터치
- 마이크: 묵죽, 달항아리, 사물놀이, 청 (4점)
- 손 인식(MediaPipe): 정음 별자리, 십이지 (2점)
- 자이로: 우중산수, 낙화 (2점, 데스크톱은 마우스 대체)
- 사운드 출력: 전 관 앰비언트(국악 오음계 기반) + 작품별 인터랙션 효과음

## 3. 기술 아키텍처

**방식**: Zero-build 바닐라 ES 모듈 (A안). 빌드 도구 없음. import map으로 three.js CDN 로딩, MediaPipe CDN 지연 로딩.

### 파일 구조

```
media-art/
├── index.html              # 단일 진입점 (아트리움 + 뷰어 셸)
├── making-of.html          # 제작 과정 페이지
├── css/style.css           # 전역 스타일 (한지 질감, 먹빛 팔레트)
├── js/
│   ├── main.js             # 해시 라우팅 + 뷰어 셸 + 작품 로더
│   ├── data.js             # 8관 32작품 메타데이터 (제목, 설명, 기법 노트)
│   ├── core/
│   │   ├── audio.js        # 공유 오디오 엔진 (Web Audio, 오음계 스케일)
│   │   ├── sensors.js      # 센서 허브 (마이크·자이로 권한/폴백 일원화)
│   │   ├── hands.js        # MediaPipe 손 인식 래퍼 (지연 로딩)
│   │   └── canvas.js       # 공통 유틸 (리사이즈, DPR, 루프, Perlin)
│   └── works/              # 작품 1개 = 파일 1개, 총 32개
│       └── w{관}-{작품슬러그}.js
```

### 작품 인터페이스 (계약)

```js
export default {
  id: 'balmuk',
  wing: 1,
  needs: [],            // 'mic' | 'hands' | 'gyro' | 'webgl' 중 필요한 것
  mount(ctx) {},        // ctx = { canvas, audio, sensors, width, height, dpr }
  unmount() {},         // 루프 정지, 리스너 해제, WebGL 컨텍스트 정리 필수
  pointer(e) {},        // 정규화된 포인터 이벤트 (mouse/touch 통합)
  resize(w, h) {},
}
```

모든 작품은 이 계약만 구현하면 된다. main.js는 작품 내부를 모르고, 작품은 서로를 모른다. → 병렬 서브에이전트 제작의 근거.

### 로딩 전략

- 초기 로딩: index.html + style.css + main.js + data.js 합계 < 100KB
- 작품 모듈: `import()` 동적 로딩 (작품 진입 시). 관 입장 시 해당 관 작품 4점 프리페치
- three.js: import map + CDN, 7·8관 일부 작품 첫 진입 시에만 로딩
- MediaPipe: 손 인식 작품(2점) 첫 진입 시에만 로딩

### UI 흐름

아트리움(전시 소개 + 관 목록) → 관 입구(작품 4점 카드) → 작품 전체화면 뷰어(‹ › 작품 이동, ⓘ 기법 노트 토글, 사운드 토글, ✕ 닫기).

해시 라우팅 `#/wing/1/work/2` — 모든 작품 URL 직접 공유 가능 (참고 사이트에 없는 개선점).

### 에러 처리·폴백 (sensors.js 일원화)

| 상황 | 처리 |
|------|------|
| 마이크/카메라 권한 거부 | 해당 작품 마우스 대체 모드 자동 전환 + 안내 문구, 작품은 계속 동작 |
| 자이로 없음(데스크톱) | 마우스 이동으로 대체 |
| WebGL 컨텍스트 실패 | Canvas 2D 간이 버전 렌더 |
| 오디오 자동재생 정책 | 첫 사용자 제스처 후 AudioContext resume |
| 작품 모듈 로딩 실패 | 뷰어에 오류 카드 표시, 다음 작품 이동 가능 |

## 4. 제작 전략

1. **코어 먼저**: index.html, css, main.js, data.js, core/* 를 단일 작업으로 완성
2. **병렬 제작**: 32작품을 관 단위(8개 에이전트)로 병렬 서브에이전트에 할당. 인터페이스 계약 고정으로 충돌 없음
3. **검증 → 배포**: 아래 3단계 검증 후 배포

## 5. 검증

1. **작품별 스모크 테스트** (Playwright): 32작품 전수 순회 — 콘솔 에러 0건, 캔버스 픽셀 샘플링으로 실제 렌더 확인, 마운트/언마운트 반복 시 루프·리스너 누수 없음
2. **성능**: 초기 전송량 < 100KB 확인, 데스크톱 60fps 목표(최소 30fps), 모바일 뷰포트(390px) 렌더 확인
3. **품질 게이트**: content-review-agent 리뷰 ≥ 85점 (레이아웃, 용어, 접근성, 오탈자) — 배포 전 필수

## 6. 배포

- 새 S3 버킷(비공개, 정적 파일) + CloudFront 배포(OAC, HTTPS 기본)
- `aws s3 sync` → CloudFront 무효화 → **배포된 URL에서** Playwright 최종 스모크 테스트
- 카메라·마이크·자이로의 HTTPS 요구사항은 CloudFront가 충족

## 7. making-of.html

이번 세션의 실제 제작 과정을 기록: 요구사항 문답 → 설계 승인 → 병렬 제작 → 검증 → 배포. 실제 사용 프롬프트 포함. 참고 사이트의 "만들고 보고 고치기" 방식과 대비되는 "설계 먼저, 병렬 제작" 방식을 보여준다.

## 8. 범위 제외 (YAGNI)

- 다국어(영문) 전체 번역 — 포인트 텍스트만 영문
- CMS/관리 화면 — 작품 추가는 data.js + works/ 파일 추가로 충분
- 방문자 통계, SNS 공유 메타 최적화 외 마케팅 기능
- 빌드 파이프라인, 프레임워크, 테스트 러너 도입
